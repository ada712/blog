    git常用注意事项

    Ada 2023/5/17

    # 分支branch

    • 新建分支
    git checkout -b feature1
    1
    • 切换分支
    git checkout  feature2
    1
    • 删除分支 （不能在本分支删除自己，需要先切换到别的分支）
    git branch -d feature1 
(出于安全考虑，没有被合并到 master 过的 branch 在删除时会失败,可以把 -d 改成-D，⼩写换成⼤写，就能删除了。)
    1
    2

    # push指令的本质

    • push 是把当前的分⽀上传到远程仓库，并把这个 branch 的路径上的所有 commits 也⼀并上传。
    • push 的时候，如果当前分⽀是⼀个本地创建的分⽀，需要指定远程仓库名和分⽀名，⽤ git push originbranch_name 的格式，⽽不能只⽤ git push；或者可以通过 git config 修改 push.default 来改变 push 时的⾏为 逻辑。
    • push 的时候之后上传当前分⽀，并不会上传 HEAD；远程仓库的 HEAD 是永远指向默认分⽀（即 master）

    # merge

    • merge 的含义：从两个 commit「分叉」的位置起，把⽬标commit 的内容应⽤到当前 commit（HEAD 所指向的commit），并⽣成⼀个新的 commit；
    • merge 的适⽤场景：
     1.单独开发的 branch ⽤完了以后，合并回原先的branch；
 2. git pull 的内部⾃动操作（先git fetch ,再git merge）。
    1
    2
    • merge 的三种特殊情况：
    1. 冲突
    原因：当前分⽀和⽬标分⽀修改了同⼀部分内容，Git ⽆法确定应该怎样合并；
    应对⽅法：解决冲突后⼿动 commit。
2. HEAD 领先于⽬标 commit：Git 什么也不做，空操作；
3. HEAD 落后于⽬标 commit：fast-forward。
    1
    2
    3
    4
    5
    6

    # rebase 在新位置重新提交

    给commit序列重新设置基础点（把你指定的 commit 以及它所在的 commit 串，以指定的⽬标 commit 为基础，依次重新提交⼀次。）

    git rebase 目标基础点

// rebase 是站在需要被 rebase 的 commit 上进⾏操作，这点和 merge 是不同的
    1
    2
    3

    如下：将dev1分支合并到master

    • 步骤一：
    git chekcout dev1
git rebase master
    1
    2

    • 步骤二:
    git checkout master
git merge dev1   // 把master移到最新的commit
    1
    2

    # 查看改动内容的方法

    • 查看历史中的多个 commit：log
    1. 查看详细改动： git log -p
2. 查看⼤致改动：git log --stat
    1
    2
    • 查看具体某个 commit：show
    1. 要看最新 commit ，直接输⼊ git show ；要看指定commit ，输⼊ git show commit的引⽤或SHA-1
2. 如果还要指定⽂件，在 git show 的最后加上⽂件名
    1
    2
    • 查看未提交的内容：diff
    1. 查看暂存区和上⼀条 commit 的区别：git diff --staged（或 --cached）
2. 查看⼯作⽬录和暂存区的区别：git diff 不加选项参数
3. 查看⼯作⽬录和上⼀条 commit 的区别：git diff HEAD
    1
    2
    3

    # git commit --amend 可修改刚提交的代码-备注信息

    commit --amend 并不是直接修改原 commit的内容，⽽是⽣成⼀条新的 commit。

    # git rebase -i 可修改之前N个的commit备注信息

    rebase -i：交互式 rebase :就是在 rebase 的操作执⾏之前，你可以指定要 rebase 的 commit 链中的每⼀个 commit 是否需要进⼀步修改。那么你就可以利⽤这个特点，进⾏⼀次「原地 rebase」。

    说明：在 Git 中，有两个「偏移符号」： ^ 和 ~。
^ 的⽤法：在 commit 的后⾯加⼀个或多个 ^ 号，可以把 commit 往回偏移，偏移的数量是 ^ 的数量。
eg：
master^ 表示 master 指向的 commit 之前的那个 commit； 
HEAD^^ 表示 HEAD 所指向的 commit 往前数两个 commit。

~ 的⽤法：在 commit 的后⾯加上 ~ 号和⼀个数，可以把 commit 往回偏移，偏移的数量是 ~ 号后⾯的数。
eg：
HEAD~5 表示 HEAD 指向的 commit往前数 5 个commit

    • 下面是示例
    • 1.执行git rebase -i HEAD~3, 或者通过git log 获取到 SHA ，执行git rebase -i xxxxxxxx也可进入下面场景
    • 2.把 pick 修改成 edit 后，就可以退出编辑界⾯了
    • 3.执行命令 git commit --amend,进行修改
    • 4.改完后，最后执行git rebase --continue,再查看是否修改成功

    # git reset --hard 丢弃最新的提交/回到哪个目标commit

    git reset --hard ⽬标commit
    1
    2
    • 额外知识-视情况选择使用
    reset 指令的本质：重置 HEAD 以及它所指向的branch 的位置。
reset 的三种参数：
1. --hard：重置位置的同时，清空⼯作⽬录的所有改动；（大白话：你的未提交的修改会被全部擦掉）
2. --soft：重置位置的同时，保留⼯作⽬录和暂存区的内容，（保留⼯作⽬录和暂存区中的内容，并把重置 HEAD 所带来的新的差异放进暂存区）
3. --mixed（默认）：重置位置的同时，保留⼯作⽬录的内容，并清空暂存区。
    1
    2
    3

    # git rebase 撤销过往的提交

    1. ⽤ git rebase -i 在编辑界⾯中删除想撤销的 commits
2. ⽤ git rebase --onto 在 rebase 命令中直接剔除想撤销的 commits
这两种⽅法，理念是⼀样的：在 rebase 的过程中去掉想撤销的commit，让他它消失在历史中

git rebase -i 的撤销可参考前面备注的修改，是类似的 只是主要是把要修改的 commit左边的 pick 改成 edit，
⽽如果你要撤销某个 commit ，做法就更加简单粗暴⼀点：直接删掉这⼀⾏就好
    1
    2
    3
    4
    5
    6

    # ⽤ rebase --onto 撤销提交

    --onto 参数后⾯有三个附加参数：⽬标 commit、起点commit（注意：rebase 的时候会把起点排除在外）、终点commit
