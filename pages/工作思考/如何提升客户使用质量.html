    质量如何提升，还有什么流程可固化

    # 质量如何提升，还有什么流程可固化

    # 业务

    系统相关人员得有全局观，减少和需求人员的信息差，多看需求多听会，尽可能串明白系统全流程

    # 质量

    开发要提升系统质量，项目启动前，梳理通用的可封装功能落地验证，并且通用功能必须由高级程序员审核封装开放，提升效率。编码过程考虑上下文影响，减少 bug

    # 信息归集

    研发测试人员要梳理常见的问题、应用部署处理、排查脚本、数据检测、测试环境造数功能、自动脚本等日常维护数据

    # 适当的冗余

    客户很重要，但是有必要建立缓冲机制，对于优先级不高的一些问题要适当的弹性解决，开发测试人员需要学习基础话术应对客户

    # 问题反馈

    系统可以有一个反馈的图标，收集平台的使用者需求，评估优化，打造更好产品

    # aio 的单处理的时候

    可以把解决方案或者预估影响范围处理意见都填写，测试验证有特殊场景的也描述验证的情况和结果

    # 强化团队沟通与协作

    组织跨职能团队会议，包括开发、测试、产品管理等，确保每个人了解整个项目的进度、目标和挑战

    # 测试人员

    建议测试人员增加自动化测试，提升效率的同时也能提升质量

    # 版本发布

    不随意发版，可和开发人员沟通此问题，避免解决一个缺陷就立刻马上发版，会在一定程度上提高测试效率

    # 对齐信息

    信息同步要及时，也要到位，避免存在信息差而导致覆盖不全面，避免信息差，重要事情需反复强调，确保相关人员收到且响应成功，即使在同一个群，> @对方之后或者即使对方已读，也可能忽略或者以为不重要，造成决策失误

    # 项目范畴：

    （1）合理排期，做好开工前的规划设计，避免因为前期设计不足导致的后续持续返工； （2）团队之前合理分工，将功能按模块拆分，尽量将需求拆分成模块，模块化开发； （3）项目成员之间交接要做好文档交接，对于复杂设计的模块，主动提出痛点告知交接方。

    # 技术范畴：

    （1）模块化：沉淀通用的且通过边界测试的功能代码，避免反复造轮子，将功能代码固化，插件化，灵活适配多种项目； （2）性能优化：完成需求之余，对功能代码做边界测试，对页面性能做分析记录，将网络性能优化纳入优化范畴，尽可能减少页面渲染耗时，避免闭包等造成内存泄漏； （3）文档化：对于复杂的需求主动补齐代码注释，项目版本记录迭代日志，项目 README；对于脚手架和组件的复杂设计也要沉淀文档； （4）增量原则：对于自己不熟悉且时间紧急的需求任务，尽量做增量迭代，不要试图去改变原有的逻辑造成严重缺陷。

    现有的流程已经比较完善了，不过严格执行下去就很难 比如开发之前的设计评审：一般是开发给出设计文档，然后评审通过之后进行开发 但是开发过程中，经常会有改动，代码改了之后，文档却很少会再去进行更新(觉得只是开发评审用的，后续用不着了，更新感觉浪费时间) 不更新对当下来说节省时间，但是后续交接就会比较麻烦 如果开发完了之后还能再输出一份完整详细设计文档，其他接手的人应该会快很多，不过要花更多的时间 对于团队来说：能解决团队成员单点故障，有问题其他人看文档也能快速上手 对于个人来说：把自己做的东西分享出去，才有时间做其他更重要的事情

    客户预期管理早日计划和安排，项目计划输出要从实际出发，预留时间缓冲时间，优化代码提升代码质量，进而提升客户体验。

    建立一个问题排查清单的文档，项目结项发版后测试研发都把自己遇到的问题、分析原因、问题定位方法和解决方案分享到这个文档，便于负责这个模块的人刚好休息，但这个模块刚好又有问题，其他同事就可以借助清单快速定位并解决问题。

    带着项目思维去做开发,设置好里程碑;重视需求评审及设计评审,与项目组成员积极沟通,不留模棱两可的疑问点,事无巨细,减少后期因设计引发的重构;各模块串联部分应主动去了解其他人负责模块的设计思路,不留坑不踩坑

    # 从开发代码层面上去看质量问题

    需求评审

    • 需求串讲，加强对需求的理解
    • 问题记录，推动提升需求质量

    系统设计

    • 系统设计评审，确保满足业务需求

    代码检查

    • 按检查项检查，重点覆盖与资金相关代码逻辑

    开发自测

    • 开发自测率100%，确保代码逻辑正确

    系统测试

    • 统一维护资金相关的测试用例，每次版本重点回归与资金相关的测试案例

    系统上线

    • 按 check list 中的检查项进行上线，确保上线过程中没有问题
